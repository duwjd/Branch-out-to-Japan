# 0. 개요

③ 운영은 3축 상품 구조의 마지막 축이다. 이 문서는 운영의 앱 메뉴 영역만 다룬다 — 브랜드 자산 라이브러리(우선), 브랜드 관리(브랜드 킷), 일본 기업 매칭(라이트), 성과 판별(자리 예약). 리뷰 축적 운영 키트·첫 4주 성과 확인 리포트 등 구독 서비스 실체는 `../03-구독플랜-재구성-제안.md`가 정본이며 이 문서 범위에서 제외한다. 운영 축의 방향 정의 정본은 `../../docs/decisions/2026-07-04-상품-구조-구체화.md` §3과 `../../07-ia.md` §4·§6이다.

화면은 브랜드 관리·자산 라이브러리·자산 상세·기업 매칭 4개로 구성한다. 사이드바 "운영"은 하위 메뉴 **[브랜드 관리 / 자산 라이브러리 / 기업 매칭]** 아코디언으로 확장되고, "운영"을 클릭하면 첫 항목인 브랜드 관리(`/app/brand`)로 진입한다 — 운영 활성 시에만 펼치고, 타 축 화면의 사이드바 마크업은 바뀌지 않는다(⓪ MAIN-01c 델타).

**운영에는 "홈" 화면이 없다** (2026-07-22 2차 개정). 구 구조는 "운영 홈이 곧 자산 라이브러리"였으나, 메뉴 순서를 사용자가 실제로 밟는 순서(**설정 → 보관 → 확장**)로 바꾸면서 첫 항목이 브랜드 관리가 됐다. 첫 항목이 곧 홈이라는 등식을 유지하면 "운영 홈 = 브랜드 관리"가 되어 라이브러리 중심이던 기존 서술과 어긋난다. 그래서 홈이라는 개념 자체를 없애고 **운영을 3화면 메뉴 그룹으로 재정의한다** — 각 화면은 사이드바 하위 메뉴에서 병렬로 진입하는 독립 화면이고, 그 위에 얹는 중간 허브 화면은 만들지 않는다(내용 없는 중간 화면을 만들지 않는다는 기존 판단은 그대로 유효하다).

메뉴 순서 근거: 브랜드 킷을 채워야 진단·생성이 정확해지고(브랜드 관리), 그 결과가 쌓이며(자산 라이브러리), 쌓인 자산을 들고 현지 기업과 연결된다(기업 매칭). 설정이 생성보다 앞서고 생성이 확장보다 앞선다.

※ **문서 섹션 순서는 메뉴 순서와 다르다** — 이 문서는 `# 1 자산 라이브러리 → # 2 자산 상세 → # 3 브랜드 관리 → # 4 기업 매칭` 순서를 유지한다. 행 번호(LIB-·DETAIL-·BRAND-·MATCH-)가 타 기획서의 참조 키라 섹션을 재배열하면 참조가 전부 깨진다. 화면 순서가 궁금하면 이 개요의 메뉴 순서를 본다.

자산 라이브러리는 신규 저장 없이 `Report`·`GeneratedAsset`을 브랜드 프로필 축으로 재조회하는 화면이다(`../../08-data-flow.md` §5.2). 콘텐츠 타입 구분(진단 리포트/썸네일)은 필터 칩이 아니라 상단 타입 탭이 담당한다 — 타입 구분 = 1차 내비, 칩·셀렉트 = 2차 정제. 콘텐츠 허브형 서비스의 표준 관례를 따른 것이다(Canva Projects — 콘텐츠 타입은 탭·사이드바, 필터는 보조 정제). ② 스튜디오가 홈=생성 퍼널로 개편되며 갤러리가 사라졌다(`../02-studio/02-studio-ui-기획서.md` # 0) — 이로써 자산 갤러리·카드 재열람의 정본 위치는 ③ 운영이다. 라이브러리는 무언가를 새로 만드는 화면이 아니라 ①·②에서 만든 자산이 모이는 화면이고, 여기서 만들지 않는다 — 카피도 그 톤으로 쓴다. 자산 카드를 클릭하면 축을 벗어나지 않고 ③ 운영 안의 자산 상세(# 2)로 들어간다. 카드 클릭으로 ②·① 축으로 이탈시키지 X — 축 이동은 자산 상세 안의 명시적 이동 동선(다른 템플릿 생성·리포트 전체 보기)에서만 일어난다.

시즌 캘린더는 별도 화면이 아니라 자산 라이브러리 상단의 시즌 타임라인 스트립으로 내장한다. 스트립은 "언제 올릴지"를 예약하는 스케줄러가 아니라 "무엇을·어떤 말로 준비할지"를 제안하는 조회 전용 영역이다 — SNS 예약 발행툴은 금지 포지션이다(`../../00-positioning.md`). 시즌 데이터 근거는 메가와리 연 4회(3·6·9·11월)·크리스마스 코프레·UV 상전·가을 신색 사이클이다(`../../research/beautyContent/일본_뷰티_인스타그램_컨텐츠_마케팅_리서치.md` §11). ⓪ 홈 KPI·"다가오는 이벤트" 위젯(⓪ MAIN-12)과 같은 데이터를 쓴다.

브랜드 관리는 생성 입력 허브다. `BrandProfile`(브랜드명·카테고리·제품분류·포지셔닝·채널·제품 메모)에 일본향 용어집·톤 가이드(브랜드 킷)를 더해 한곳에서 편집한다. 저장값은 ① 진단 입력 프리필과 ② 생성 참조의 소스이고, 진단 원본은 `tierInput` 스냅샷이므로 킷 수정은 발행된 리포트에 소급되지 X(`../../08-data-flow.md` §6). 편집 정본은 이 화면 하나다 — 마이페이지 브랜드 프로필 섹션은 조회 요약만 남기고 "수정"이 이 화면으로 연결된다(계정 MYPAGE-06 델타). 참고 관례: Canva Brand Kit — 프로필·보이스 허브를 생성 도구가 자동 참조. 단, 생성 자동 참조 계약은 제안 상태다(BRAND-09). 브랜드 킷은 일본 고객 관점 메시지 재설계의 입력이지 범용 번역 사전이 아니다 — 용어집·금지 표현은 근거(약기법·검수 게이트)와 함께만 다룬다.

기업 매칭은 컨시어지형 라이트 화면이다. 매칭 DB 열람·조건 검색 없이 신청 폼과 상태 추적(신청 완료→검토 중→제안 도착)만 둔다. 상태 갱신은 운영팀 수동이다. 매칭 입출력 계약은 미정의(`../../08-data-flow.md` §5.2 TBD)라 본 문서는 화면 요구만 확정하고, `MatchRequest` 엔티티 신설 제안을 기타 행(MATCH-08)에 남긴다.

앱 셸(사이드바 전역 내비·브랜드 스위처·KPI 위젯)은 ⓪ 메인 기획서 MAIN-01을 정본으로 따른다. 이 기획으로 사이드바 "운영" 메뉴가 활성화된다("준비 중" 비활성 → "라이브러리 우선 △" + ③ 브랜드 관리 이동 — 2026-07-22 착지 변경) — ⓪·①·②·계정 앱 화면의 사이드바 델타가 함께 발생한다. 그리드의 생성중 타일은 ② 기획서가 제안한 `GeneratedAsset.status` 델타를 전제한다. PC 기준으로 설계하며 모바일은 반응형 대응만 한다. 성과 판별은 실현가능성·우선순위 미정(2026-07-04 결정)이라 자리 예약 카드만 두고 지표·수치 UI로 그리지 않는다(증거 원칙).

# 2. 자산 상세

운영 라이브러리 안에서 자산 1건을 다시 열어 보는 화면이다. 운영 = 자산 보관·재열람의 정본 위치이므로, 카드 클릭 시 ②·① 축으로 이탈하지 않고 이 화면으로 들어온다. 여기서 만들지 않는다 — 생성은 ②, 리포트 전체 열람은 ① 몫이며 축 이동은 화면 안의 명시적 동선에서만 일어난다. 자산 종류에 따라 두 모드로 갈린다: 썸네일 자산 = 썸네일 모드(DETAIL-01~04), 리포트 자산 = 리포트 요약 모드(DETAIL-05). DETAIL-06·07은 두 모드 공통이다. ② 결과 상세(RESULT-)의 표기 문법을 축약 준용하되 재생성·편집 액션은 두지 X(조회·재열람 전용).

# 3. 브랜드 관리

브랜드 킷을 편집하는 화면이다. `BrandProfile` 편집 정본은 이 화면 하나다 — 온보딩과 브랜드 추가 모달(⓪ MAIN-01b′)은 최초 생성만, 마이페이지는 조회 요약만 담당한다(계정 MYPAGE-06 델타). 저장값은 ① 진단 입력 프리필·② 생성 참조의 소스이고, 진단 원본은 `tierInput` 스냅샷이므로 킷 수정은 발행된 리포트에 소급되지 X(`../../08-data-flow.md` §6). 프로필·제품 자산·채널·일본향 용어집의 4개 섹션 카드로 구성한다. 여기서 자산을 만들지 않는다 — 생성 진입 동선을 두지 X.

**2026-07-22 개편 — 제품 자산 CRUD.** 구 BRAND-03은 제품 정보를 텍스트 메모 1개로 받았다. 브랜드가 제품을 여러 개 파는데 킷은 그걸 구분하지 못했고, ② 스튜디오는 생성할 때마다 같은 제품컷을 다시 업로드하게 했다. 이번 개정으로 제품을 **레코드 단위로 등록**하고 제품당 **제품컷 이미지를 여러 장** 보관한다. 등록한 제품컷은 ② 스튜디오 홈에서 업로드 대신 골라 쓰는 입력이 된다(`../02-studio/02-studio-ui-기획서.md` HOME-02 — 2026-07-22 소스 선택 신설). 브랜드 킷이 "적어 두는 곳"에서 "생성이 실제로 집어 가는 곳"이 되는 개정이다.

제품 편집을 모달로 두는 근거: 입력 항목이 7개에 이미지 업로더까지 붙어 카드 안에서 인라인으로 펼치면 그리드가 크게 밀린다. 생성·수정은 시작과 끝이 분명한 단발 작업이라 모달이 맞다. 대신 삭제는 되돌릴 수 없으므로 확인 모달을 따로 둔다(BRAND-03c).

# 0. 개요

메인(앱 홈)은 로그인·온보딩(브랜드 프로필 등록) 후 진입하는 앱 홈이자 3축 허브 화면이다. 브랜드 프로필 컨텍스트 위에서 브랜드 자산 요약 · 진행중 작업 · 다음 단계 CTA 3요소를 한 화면에 모으고(`../../07-ia.md` §4), ① 진단 리포트 · ② 마케팅 스튜디오 · ③ 운영으로 가는 진입점이 된다. 사이드바 메뉴명은 **"홈"**이다(2026-07-22 — 구 "대시보드"). 지표를 나열하는 화면이 아니라 브랜드의 현재 상태와 다음 할 일을 보여 주는 화면이라 이름도 그렇게 부른다.

셸은 좌측 사이드바 + 본문이다. 사이드바가 브랜드 스위처·3축 내비·KPI 위젯·품의 PDF·계정을 담고, 본문은 다음 단계 밴드 → 위젯 그리드 → 최근 자산 순으로 쌓는다 — 진행중 작업은 우하단 플로팅 패널로 분리한다. 이 사이드바 셸은 앱 전역 공통이며 ①·② 앱 화면도 동일 셸을 쓴다(07-ia §5 "사이드/탭에 3축"). 화면은 메인 1개로 구성하고, 별도 화면 대신 상태 변형(첫 방문·기본·진행중·로딩·오류)이 화면 구성을 바꾼다.

설계 원칙은 지표 나열형 대시보드가 아니라 "다음 단계 제안"이다 — 3축 성숙도가 서로 달라(① 확정 v4 · ② 썸네일만 확정 · ③ 방향만) 사용자가 다음에 무엇을 할지 모르는 상태가 기본값이기 때문이다. 유사 서비스 7종 조사와 적용 결정은 `../../research/main-dashboard-ui-references.md`. 화면에는 시스템 카운트·발행 산출물만 출력하고 시장 성과 수치·리뷰 등 근거 없는 데이터는 출력하지 않는다(증거 원칙).

**2026-07-22 개편의 논리** — 다음 단계 제안 하나만 남긴 구 구성은 "무엇을 할지"에는 답했지만 "지금 내 브랜드가 어떤 상태인지"에는 답하지 못했다. 그 답이 사이드바 KPI 4줄 숫자에만 있어서, 진단 결과를 다시 보려면 ①로, 브랜드 킷 상태를 보려면 ③으로 나가야 했다. 이번 개정은 히어로를 컴팩트 밴드로 낮추고 그 아래에 **리포트 요약(MAIN-10) · 브랜드 정보(MAIN-11) · 다가오는 이벤트(MAIN-12)** 위젯을 둔다. 세 위젯 모두 **기존 엔티티 재조회 전용**이다 — 홈은 아무것도 새로 저장하지 않는다(`../../08-data-flow.md` §7). 위젯은 각 축의 요약일 뿐이며 상세는 각 축이 담당한다: 리포트 전체는 ①, 브랜드 킷 편집은 ③ 브랜드 관리, 시즌 전체 타임라인은 ③ 자산 라이브러리다. 홈에서 편집·생성하지 X.

추천은 여전히 화면당 1개다. 위젯이 늘어도 primary 버튼은 다음 단계 밴드의 CTA 하나뿐이고, 위젯의 이동 동선은 전부 secondary·텍스트 링크로 둔다.

이 화면의 진행중 작업 표시는 ② 기획서가 제안한 데이터 계약 델타(`GeneratedAsset.status` 또는 잡 조회, `../02-studio/02-studio-ui-기획서.md` # 0)를 전제한다. 리포트 처리 상태는 `DiagnosisRequest.status`를 읽는다(`../../08-data-flow.md` §7).

PC 기준으로 설계하며 모바일은 반응형 대응만 한다(사이드바 → 상단 블록 스택). 온보딩, 3축 내부 화면(①·② 기획서), ③ 운영 상세는 이번 범위에서 제외한다. 로그인·마이페이지는 별도 기획서(`../03-account/03-account-ui-기획서.md`)에서 다루고, 이 화면은 사이드바 계정 메뉴로 연결만 한다. 상위 구조 정본은 `../../07-ia.md` §4·§5, 화면↔데이터 매핑은 `../../08-data-flow.md` §7이다. 구세대 명세 `design/wireframes/app-spec.md`의 요소 인벤토리를 계승하되, 본 문서가 화면 동작·상태·예외처리의 정본이 된다.

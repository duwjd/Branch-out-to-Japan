# 0. 개요

온보딩은 방금 가입해 **브랜드·리포트·썸네일이 하나도 없는 유저**를 첫 진단까지 이어 주는 first-run 구간이다. 대상은 로그인 직후의 신규 유저다. 목 인증(데모 유저 1명·브랜드 싱글턴 `id='default'`)이라 "신규 유저"는 실무상 **브랜드 프로필 미등록**(`getBrandProfile() == null`) 상태와 같다. 온보딩 트리거는 가입 이벤트가 아니라 이 no-brand 조건이다. (2026-07-23) 비회원도 이 캡처 화면을 열람·작성할 수 있으며, 첫 브랜드를 저장하는 시점에 로그인이 유도된다(ONBOARD-00·02 · `../03-account/03-account-ui-기획서.md` §3 GATE).

모델은 **하이브리드**다. 가입 직후 초경량 첫 브랜드 캡처 1화면(필수 3필드)으로 3축이 공유하는 공통 엔티티(브랜드 프로필)만 먼저 만들고, 나머지(제품·진단·썸네일)는 홈 4단계 셋업 가이드(⓪ MAIN-06)가 점진 유도한다. 전용 위저드(`../../09-dev-spec.md` §2 `/onboarding` — 보류)는 만들지 않는다. 캡처 화면은 별도 라우트가 아니라 `/app`의 no-brand 본문 상태로 둔다 — 목 인증이라 가릴 하위 화면이 없어 스크림 모달은 맞지 않고, `/app`은 이미 브랜드 프로필을 조회한다.

캡처 필드 3종의 정본은 ⓪ MAIN-01b′ 브랜드 추가 모달과 동일하다 — 같은 값을 두 벌로 정의하지 X. MAIN-01b′는 **2번째 이후 브랜드**를 만드는 모달이고, 이 문서의 캡처는 **최초 1개 브랜드**를 만드는 first-run 경로다. 두 경로가 같은 3필드·같은 검증을 쓴다. 상세 입력(제품·채널·용어집)은 캡처가 아니라 ③ 브랜드 관리(BRAND-00)가 담당한다. 정본 구조는 `../../07-ia.md` §4·§8, 홈 상태 변형은 `00-main-ui-기획서.md` MAIN-06·MAIN-13이다. PC 기준으로 설계하고 모바일은 반응형 대응만 한다.

# 2. 상태 커버리지 매트릭스

앱 화면이 사용자 상태별로 무엇을 출력하는지의 기준표다. 상태 드라이버는 `brand`(브랜드 프로필 유무) · `report`(발행 리포트 유무) · `thumb`(완성 썸네일 유무) · `job`(진행 중 잡 유무)다. 구현 표기 — 구현: 이미 있음 · 부분: 있으나 보강 필요 · 신규: 새로 만듦. 여기에 **인증 상태(비회원/회원)**가 겹친다(2026-07-23) — 비회원은 각 화면을 그대로 열람하되 생성·등록·저장 액션에서 로그인 유도 게이트가 걸린다(`../03-account/03-account-ui-기획서.md` §3 GATE). 아래 데이터 상태 행은 회원 기준이고, 비회원 오버레이는 첫 행에 요약한다.

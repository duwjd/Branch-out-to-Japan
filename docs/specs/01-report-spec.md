> ## ⚠ v4 개정 (2026-07-16) — 입력 브랜드 우선 재구성 · 두 진단 모드
> **1. 진단의 주어가 브랜드로 바뀐다.** 입력은 **브랜드 섹션(필수: 카테고리·브랜드명·포지셔닝)** 과 **제품 섹션(전부 선택: 분류·제품명·성분·가격·상세페이지 콘텐츠)** 의 2단 구조다(§3 전면 개정). **상세페이지 콘텐츠도 선택**이 됐다 — 이전의 "Tier 0/1/2" 어휘는 은퇴한다(분할선이 옛 티어 경계를 가로지름).
> **2. 산출물이 두 모드로 갈린다.** `브랜드 진단`(콘텐츠 없음)과 `브랜드+제품 진단`(콘텐츠 있음). 블록 1(종합점수)·3(약기법 감사)·5(A~E 채점)·7(재작성)·8(비포&애프터)은 고객 문장 위에서 계산되므로 **브랜드 진단에서는 "데이터 잠금"** 상태로 출력된다("상세페이지 카피를 넣으면 열립니다" + 무엇이 열리는지 명시). 블록 0·2·6·9와 블록 4의 코퍼스 측은 브랜드 진단에서도 완전 산출. **근거 없는 대체 수치(예: "브랜드 포지셔닝 점수")를 발명하지 않는다** — 루브릭·코퍼스 없는 숫자는 증거 원칙 위반.
> - 본문 §4의 블록 서술과 §1 목표는 **브랜드+제품 진단 기준**으로 읽는다.
> - **불변식 재서술:** "감사표 없는 리포트는 발행하지 않는다" → **"제품 콘텐츠가 제출된 진단은 감사표 없이 발행하지 않는다."** 브랜드 진단에는 감사 대상 문장 자체가 없다.
> - 블록 4 대비표의 "내 콘텐츠" 칸은 브랜드 진단에서 **"미확인"**(제3의 값)이다 — "미관찰"은 *찾아봤는데 없었다*는 주장이므로 콘텐츠 없이 쓰면 증거 원칙 위반.
> **3. 가격·무료 경계(§2 개정).** **두 모드 모두 유료다. 무료는 "샘플"뿐** — 제출 즉시 샘플(블록0 표지 전체 + 블록1 요약 일부(종합점수 가림) + 블록2 페르소나 카드 맛보기)은 열람되고, 풀 리포트는 결제 후 열람. 브랜드 풀 진단 가격 **(미정)**. 결제 게이트 자리 = 샘플 → 풀 열람 직전(08 §8-D2). **MVP는 결제 집행 미구현** — 30만 진단이 실결제 없이 열람되는 현 상태와 동일하게, 샘플 경계도 정의만 하고 집행하지 않는다.
> **4. 보고용 슬라이드(§10)도 모드를 따른다.** 브랜드 진단의 덱은 7장이 아니라 **4장**(표지 · 포지셔닝/USP · 벤치마크 · 다음 단계) — 측정하지 않은 것을 결재자 앞에 놓지 않는다.
>

## 1. 목표 · 비목표

### 하는 것 (목표)
1. 한국 뷰티 브랜드의 **KR 상세페이지 콘텐츠**를 입력받아, 일본 시장 관례(라쿠텐 코퍼스 근거) 대비 **무엇이 부족한지**를 A~E 루브릭으로 채점한 **일본 문법 진단 점수(A~E 원점수 + 종합 0~100)**를 산출한다.
2. **약기법(薬機法) 표현 전수 감사**를 정식 섹션으로 제공한다 — 고객 문장별로 **【불가】/【조건부】/【가능】** 을 판정하고, "왜(조항 각주)"와 **소구력을 유지한 합법 대체 표현**을 함께 제시한다. 이는 **문장 재설계 관점의 1차 스크리닝**이며 법적 확정 판정이 아니다(면책·실명 검수자 서명 동반, §4-블록3).
3. **카테고리 경쟁 벤치마크** — 고객의 `category` 코퍼스 실측 표현을 인용해 고객 카피와 대비하고, 검색 친화 표기 전환을 제안한다.
4. 일본 고객 관점의 **타깃 페르소나·구매여정(구매 반대 이유 포함)** 과 **USP 재정의**(한국식 소구 → 일본 구매이유)를 제시한다.
5. **리뷰(口コミ) 인과 서사** — "어떤 정보 공백이 어떤 불신·이탈로 이어지는가"를 카테고리 리뷰 패턴 근거로 설명한다(하이브리드, §4-블록6).
6. 저점 항목을 근거로 **NG/OK 재작성(Before/After) + 한국어 병기**와 **비포&애프터 샘플 1종**을 제시한다.
7. 모든 산출에 **채점 기준·코퍼스 근거·규정 출처를 함께 노출**해 "번역이 아닌 재설계"를 측정 가능하게 만든다(신뢰 장치).

### 안 하는 것 (비목표 — 금지 포지션 경계)
- **번역 서비스 아님**: 문장을 일본어로 옮기는 게 목적이 아니다. 채점·재작성은 "정보 구조(근거 라벨·등급·무첨가·성분 정량·서사)"를 진단·재설계하며, 번역 품질을 매기지 않는다(루브릭 §4 말미 원칙). → 금지 포지션 5(번역).
- **종합 대행 아님**: 물류·통관·인허가·광고 집행·계정 운영을 하지 않는다. 리포트는 **사전 진단 문서**에서 끝난다. → 금지 포지션 6(종합 대행).
- **약기법 "법판정"은 아니되 "문장 감사"는 정식 제공(v2 정교화)**: 리포트의 약기법 감사는 위법 여부의 **법적 확정 판정을 내리지 않는다.** 대신 **문장 재설계 관점의 1차 스크리닝**(불가/조건부/가능 + 조항 각주 + 합법 대체 표현)을 **정식 섹션으로 제공**한다. 면책 문구·상장 전 전문가 확인 권고·실명 검수자 서명을 반드시 동반한다. 조항 DB 매칭형 정밀 자동 검출은 Free 체커/v2 영역. (이전 스펙의 "약기법 법판정 배제 = 리포트는 힌트만"을 **"법판정은 아니되 문장 감사는 정식 제공"** 으로 정교화 — §9-Q4 참조)
- **가상 성과 약속 아님**: "매출 N% 상승" 같은 예측·보장 문구 금지. 종합점수 0~100은 **일본 상세 관례 충족도**이지 성과 예측이 아니다. 벤치마크·리뷰 수치는 **코퍼스 실측치만** 사용하고, 가짜 리뷰·가짜 통계·익명 성공 사례는 생성하지 않는다(증거 원칙).

## 3. 입력 스펙 (브랜드 우선 2단 구조 — v4 전면 개정)

원칙: **진단의 주어는 브랜드다. 브랜드 정보만으로도 리포트가 끝까지 나온다**(브랜드 진단). 제품 상세페이지 콘텐츠는 선택적 심화 입력이며, 넣으면 약기법 전수 감사·A~E 채점·재작성이 열린다(브랜드+제품 진단). "**정밀도 게이지**" 원칙은 유지 — 넣을수록 **카테고리 일반형 → 우리 브랜드/제품 실측형**으로 정밀해진다. 선택 필드는 **콘텐츠 창작을 요구하지 않으며(사실만 입력), 가짜값은 금지**한다(증거 원칙).

> **v2의 "Tier 0/1/2" 어휘는 은퇴.** 이번 분할선(브랜드 필수 / 제품 선택)이 옛 티어 경계를 가로지른다 — `brandName`은 Tier 1→필수로, `source`는 Tier 0→선택으로 이동했다. 저장 키 `tierInput`은 역사적 이름으로 유지한다(리네임 비용 대비 이득 0).

### 3.1 브랜드 섹션 (전면 노출)

| 필드 | 필수 | 타입 | 검증 규칙 | 비고 |
|---|---|---|---|---|
| `brandName` 브랜드명 | **필수** | text | 1~60자 | 표지·페르소나·USP·슬라이드 파일명의 주어 |
| `positioning` 브랜드 포지셔닝 | **필수** | `{ tags[], note }` | tags: 택소노미에서 1~5개 선택 / note: 자유 서술 0~500자 | **v4 신규.** 콜③(페르소나·USP 재정의)의 핵심 입력 — [[../08-data-flow]] §4.3이 요구해 온 "포지셔닝 태그"의 실제 수집 지점. 택소노미는 실재 자산(sns-lexicon·메시지 관례 루브릭·페르소나 리서치)에서 도출하며 창작 금지 |
| `category` 카테고리 | **필수** | enum | `skincare`/`makeup`/`suncare`/`cleansing` 중 1 | E군 가중·카테고리 코퍼스·벤치마크의 키. 콘텐츠 없이도 살아남는 유일한 엔진 키라 브랜드 섹션에 둔다 |
| `targetMemo` 타깃/고민 메모 | 선택 | textarea | 0~500자 | 페르소나 생성 보조 입력(콜③) |

### 3.2 제품 섹션 (접힘 · 전부 선택)

| 필드 | 타입 | 검증 규칙 | 없을 때 폴백 |
|---|---|---|---|
| `source` 상세페이지 콘텐츠 | oneOf(`url`\|`text`) | 제출 시: `url`은 http(s) 도달 가능 / `text`는 하드 게이트 **50자 이상** | **미제출 = 브랜드 진단**(§3.3). `url` 스크래핑 실패 시 `text` 붙여넣기를 1급 경로로 안내 + "제품 정보 없이 브랜드 진단만 받는" 출구 병기 |
| `productClass` 제품 분류 | enum | `화장품`/`의약외품`/`미상` | 미입력 → `미상` 정규화. 콘텐츠가 **있을 때** 미상이면 `화장품` 가정 + "분류 미확인" 경고(약기법 감사 판정 프레임 키, §4-블록3). 콘텐츠가 없으면 감사가 없으므로 가정 경고도 없다 — 표지에 "—(제품 정보 미입력)" |
| `productName` 제품명 | text | 1~120자 | 콘텐츠에서 추출 시도, 없으면 "(제품명 미기재)" |
| `keyIngredients` 핵심 성분 | text[] | 각 1~40자, 최대 8개 | D군 채점·벤치마크는 콘텐츠 내 성분 표현 기준으로만. **브랜드 진단에서는 미사용**(콜①이 돌지 않음 — 폼에 명시) |
| `priceJPY` 예상 판매가(엔) | number | ≥0 | 페르소나 가격 감도 추론에서 제외 |

> `source`는 **url 또는 text 중 1개**를 받는 단일 필드(내부 저장은 `sourceType`/`sourceUrl`/`sourceText`로 정규화). **텍스트 길이 이중 기준(확정 유지):** 50자 = 하드 게이트(미만이면 제출 차단 — 단, 아예 비우면 브랜드 진단으로 제출 가능), 200자 = 소프트 품질선(50~199자는 생성되나 "정밀도 제한" 배지). 게이트 규칙의 단일 정의 = `lib/engine/rules/gates.ts`.

### 3.3 진단 모드 (제출 경계에서 확정)

| 조건 | 모드 | 산출 |
|---|---|---|
| `source` 미제출 | **`brand` 브랜드 진단** | 블록 0·2·4(코퍼스 측)·6·9 완전 산출. 블록 1·3·5·7·8은 **데이터 잠금**("상세페이지 카피를 넣으면 열립니다"). 종합점수 없음 — 대체 수치 발명 금지 |
| `source` 제출 | **`brandProduct` 브랜드+제품 진단** | 9블록 전부 (v2~v3 서술 그대로) |

- 모드는 **제출 시점에 서버가 한 번 판정해 `tierInput` 스냅샷에 물질화**한다(`mode` 필드) — 발행된 리포트의 입력은 자기 서술적이어야 한다.
- 제품 필드 일부만 있고 `source`가 없어도 **에러가 아니다** — 브랜드 진단으로 가되 입력된 사실은 스냅샷에 담는다.
- URL 스크래핑 실패는 브랜드 진단으로 **조용히 강등하지 않는다** — 사용자가 제품 진단을 명시적으로 요청했으므로 실패 처리 + 텍스트 붙여넣기/브랜드 진단 두 출구를 안내한다.

### 3.5 최소 입력 폴백 규칙 (반드시 구현)
- **최소 성립 조건:** 브랜드 섹션 필수 3종(`brandName` + `positioning` + `category`). 이 조건만 충족하면 리포트를 끝까지 생성한다(브랜드 진단 — 데이터 잠금 블록 포함).
- 선택 필드 결측 시 **해당 근거의 채점·생성을 건너뛰거나 콘텐츠 추론·카테고리 일반형으로 대체**하되, 리포트에 "이 입력이 있으면 더 정밀해집니다"(정밀도 게이지) 안내를 표기. **가짜값으로 채우지 않는다**(증거 원칙).
- **KR 상세는 이미지 위주가 많다**는 현실 리스크: `url` 실패·텍스트 빈약 시 `text` 붙여넣기를 1급 경로로 안내한다(§9-Q1, 이미지 OCR은 v2).

### 블록 5. 일본 문법 진단 점수 (A~E 루브릭 — 결정적 엔진)
- **채점 원본:** `research/jp-detail-message-patterns.md` §4. 항목 A1~A5·B1~B2·C1~C3·D1~D4·E1~E4를 **각 0(없음)/1(부분)/2(관례 충족)** 로 채점.
- **표시:** **A~E 그룹 막대**(축별 원점수 병행 노출) + **항목별 아코디언**. 각 항목 아코디언은 `점수 · 통과기준 · 내 문장 인용 · 코퍼스 근거` 4요소 노출(채점기준 공개 원칙 — 점수만 던지지 않는다).
- **집계·가중 (MVP 확정 공식, 불변):**
  - 그룹 점수(%) = (그룹 내 항목 점수 합) ÷ (적용 항목 수 × 2) × 100.
  - **E군은 제품 카테고리에 해당하는 항목만 채점**(suncare→E1, makeup→E2, cleansing→E3, skincare→E4). 나머지 E항목은 N/A로 제외(분모에서도 제외).
  - **종합 점수(0~100)** = Σ(그룹 점수 × 그룹 가중치). 가중치는 카테고리별(아래 표, 합=1.00). **헤더(블록1)의 0~100은 이 값**.

  | 그룹 | skincare | suncare | makeup | cleansing |
  |---|---|---|---|---|
  | A 신뢰 구축 | 0.30 | 0.20 | 0.15 | 0.20 |
  | B 무첨가·안전 | 0.15 | 0.10 | 0.10 | 0.30 |
  | C 서사 구조 | 0.25 | 0.10 | 0.20 | 0.15 |
  | D 성분 프레이밍 | 0.20 | 0.20 | 0.15 | 0.15 |
  | E 카테고리 적합성 | 0.10 | 0.40 | 0.40 | 0.20 |

  > 가중치는 루브릭 §2-d·판정 가이드에 근거한 **초기값**. 파일럿 튜닝 대상(§9-Q2).
- **결정성 계약:** 같은 항목 점수 → 같은 종합 점수(집계·가중은 코드). 밴드 라벨 임계값은 §9-Q3(보류).

## 5. 산출 로직 (개발자용)

각 섹션을 무엇으로 생성하는가 — **규칙기반(루브릭·정규식) vs Claude 생성**을 분리한다.

### 5.1 파이프라인 개요
```
입력 → [1 콘텐츠 수집/정규화(규칙)] → [2 사전 신호 추출(규칙)]
     → [콜① 루브릭 채점(LLM)] → [3 집계·가중(규칙, 결정적)]
     → [콜② 약기법 전수 감사(LLM)]
     → [콜③ 페르소나·USP·리뷰 서사(LLM)]
     → [4 벤치마크 대비(MVP=규칙 흡수 / 목표=콜⑤ LLM)]
     → [콜④ NG/OK 재작성 + KR병기 + 샘플(LLM)]
     → [5 리포트 9블록 조립·렌더(규칙)]
```

### 5.2 LLM 콜 정의 (MVP 4콜 / 목표 5콜)

| 콜 | 방식 | 블록 | 입력 | 출력 | grounding |
|---|---|---|---|---|---|
| **① 루브릭 채점** | LLM(구조화 JSON) | 5 | 정규화 평문 + 사전 신호 + `category`/`productClass` + 루브릭 §4 전문 + 코퍼스 벤치마크 요약 | 항목별 `{score:0/1/2, evidenceQuote, criterionRef, corpusRef}` | 루브릭 항목·통과기준, 코퍼스 사전집계 요약, 렉시콘 상위 어휘 |
| **② 약기법 감사** | LLM(구조화 JSON) | 3 | 정규화 평문(문장 분해) + `productClass` + **규정 출처 요약(조항 프레임)** | 문장별 `{판정: 불가/조건부/가능, 근거, 조항각주ref, 합법대체표현JP}` | 규정 출처 요약(56항목·適正広告GL·景表法). 정밀 조항 DB는 v2 |
| **③ 페르소나·USP·리뷰 서사** | LLM(구조화 JSON) | 2·6 | 콘텐츠 요약 + `category`/`priceJPY`/`targetMemo` + 카테고리 코퍼스 패턴 + 페르소나 여정 원칙 + 리뷰 패턴 일반형 | 페르소나 카드 + 구매 반대 이유 + USP 표 + 리뷰 인과 서사(JSON) | 카테고리 관례(makeup=퍼스널컬러 등), 구매 여정(인스타→리뷰→전환), 카테고리 리뷰 패턴 일반형 |
| **④ NG/OK + KR병기 + 샘플** | LLM(구조화 JSON) | 1·7·8 | 저점 항목(콜①·집계) + 약기법 불가/조건부 문장(콜②) + 문제 문장 + 코퍼스 재설계 예시 + 렉시콘 | 헤더 총평 + NG/OK 3~5쌍(After JP+KR역문+이유) + 샘플 1종 | 코퍼스 실측 표현(`効能評価試験済み`·`○○フリー`·`6%`·랭킹 표기), 렉시콘 실제 JP 어휘 |
| **⑤ 벤치마크(목표)** | LLM(구조화 JSON) | 4 | `category` 코퍼스 서브셋 인용 + 고객 문장 | 코퍼스 인용·대비표·검색표기 전환(JSON) | **MVP에서는 규칙 코드(사전집계 요약 + 정규식 대비)로 흡수**, 문장화는 콜④가 겸함. 목표 단계에서 독립 콜로 분리 |

- **LLM 콜 수:** **MVP 4콜**(①②③④). 벤치마크(블록4)의 대비·인용은 **MVP에서 사전집계 규칙 + 콜④ 문장화로 흡수**하고, **목표 5콜**에서 벤치마크를 독립 콜⑤로 분리해 깊이를 올린다.
- **구조화 출력:** 모든 콜은 스키마 고정 JSON(파싱 안정성). 프롬프트에 **"코퍼스·규정 근거 밖의 수치·인증·리뷰를 창작하지 말 것"** 을 명시(증거 원칙 가드레일).
- **grounding 주입 형태:** 원본을 통째로 넣지 않고 **카테고리별 사전집계 벤치마크 4종**(skincare/makeup/suncare/cleansing 상위 배지·성분·카테고리 문법 표) + **규정 출처 요약**(약기법 콜②용)을 프롬프트 컨텍스트로 주입. **리뷰 실측 크롤링·RAG 검색형은 v2.**
- 구현 규칙(camelCase·JSDoc·로거·타입 안전·`any` 지양·접근성)은 **구현 시 CLAUDE.md 준수**. 본 스펙은 로직 계약만 정의.

### 5.3 결정성 계약 (보존 — 핵심)
- **규칙기반(결정적):** 콘텐츠 파싱(1)·사전 신호 정규식(2)·**집계·가중·Top3(3)**·9블록 조립(5)·**슬라이드 렌더(§10)**. **점수 계산은 결정적** — 같은 항목 점수 → 같은 종합 점수(단위 테스트로 보증).
- **LLM(판단·생성):** 항목 0/1/2 판정(콜①)·약기법 판정(콜②)·페르소나/USP/리뷰(콜③)·재작성/샘플(콜④). LLM은 **점수를 매기되 가중·합산은 하지 않는다**(집계는 코드).
- **점수 엔진과 약기법 판정 분리(불변):** 약기법 감사(콜②)는 **점수 엔진과 독립**이다. 채점에는 **A3(의약외품 등급) 등급 힌트만** 반영되고, **전수 감사표(블록3) 자체는 정성 섹션**으로 점수에 합산되지 않는다. 이로써 "약기법 판정이 흔들려도 종합점수는 결정적"이라는 계약을 지킨다.

## 6. 유저 스토리 + 수용 기준(AC)

### US-1 (S1) 최소 입력으로 진단
"일본 진출을 준비하는 브랜드 담당자로서, 우리 브랜드가 무엇인지만 알려줘도 진단 리포트를 받고 싶다. 상세페이지까지 넣으면 문장 단위 감사·점수까지 받고 싶다. 왜냐하면 대행사 견적 전에 빠르게 통할지 확인하고 싶기 때문."
- **AC-1.1**(v4 개정) 브랜드 섹션 필수 3종(`brandName`+`positioning`+`category`)만으로 리포트가 끝까지 생성된다(브랜드 진단 — 블록 1·3·5·7·8은 데이터 잠금 상태로 출력). `source`까지 제출하면 9블록 전부 생성된다(브랜드+제품 진단).
- **AC-1.2** 선택 필드 결측 시 가짜값을 넣지 않고, "정밀도 게이지" 안내와 함께 가능한 블록까지(카테고리 일반형·데이터 잠금 포함) 산출한다.
- **AC-1.3** `source=url` 스크래핑 실패 시 텍스트 붙여넣기 폴백 경로를 에러 메시지로 안내한다(+ 브랜드 진단 출구 병기). 브랜드 진단으로 조용히 강등하지 않는다.
- **AC-1.4** 제출된 텍스트가 50자 미만이면 **하드 게이트로 생성 비활성**, 50~199자는 생성되되 **"정밀도 제한" 배지**가 붙는다. 텍스트를 아예 제출하지 않으면 게이트가 발동하지 않고 브랜드 진단이 된다.
- **AC-1.5**(v4 신규) 브랜드 진단의 데이터 잠금 블록은 잠긴 이유("고객 문장이 없어 산출 불가")와 여는 방법("상세페이지 카피 입력")을 명시하고, **근거 없는 대체 수치를 표기하지 않는다**. 블록 4 대비표의 "내 콘텐츠" 칸은 "미확인"으로 표기한다("미관찰" 금지 — 찾아본 적이 없다).

### US-2 (S1) 문법 점수를 근거와 함께 확인
"담당자로서, 점수뿐 아니라 왜 그 점수인지(내 문장 어디가·일본 관례는 어떤지) 보고 싶다. 왜냐하면 근거 없는 AI 점수는 못 믿기 때문."
- **AC-2.1** A~E 전 항목이 0/1/2로 채점되고, 각 항목 아코디언에 `통과기준·내 문장 인용·코퍼스 근거`가 노출된다.
- **AC-2.2** **종합 점수는 0~100으로 헤더에 표기**되고, 상세는 **A~E 축별 원점수**를 병행 노출한다. 종합 점수는 §블록5 가중 공식으로 계산되며, **같은 항목 점수 입력 → 같은 종합 점수**(결정적)임을 단위 테스트로 확인한다.
- **AC-2.3** E군은 제품 카테고리에 해당하는 항목만 채점·분모 반영된다.
- **AC-2.4** 종합 점수 옆에 밴드 라벨(시급/보완/양호)은 **표기하지 않는다**(§9-Q3 보류) — 숫자·총평·A~E만.
- **AC-2.5** 리포트 어디에도 코퍼스·규정 실측 밖의 가상 통계·익명 성공 사례·성과 보장 문구·가짜 리뷰가 없다(증거 원칙 — QA 체크).

### US-3 (S1/S4) 고칠 문구 예시를 받는다 (한국어로 검수 가능)
"담당자로서, 저점 항목에 대해 '이 문장을 이렇게 바꾸라'는 Before/After를 받되 **일본어를 못 읽어도 무엇이 바뀌었는지** 알고 싶다. 왜냐하면 구체적 예시 + 한국어 확인이라야 신뢰하고 실행할 수 있기 때문."
- **AC-3.1** NG/OK 3쌍 이상이 산출되고, 각 쌍이 특정 루브릭 항목 또는 약기법 판정과 연결된다.
- **AC-3.2** 각 After(JP)에 **한국어 역문**이 병기되고, "무엇을·왜 바꿨나" 이유가 명시된다.
- **AC-3.3** 비포&애프터 샘플 1종이 KR 병기와 함께 포함된다. 고객 콘텐츠 빈약 시 데모 샘플은 "예시(데모)" 라벨이 붙는다.
- **AC-3.4** After는 근거 라벨·각주·무첨가·성분 정량 중 무엇을 더했는지 명시하고, 블록2 USP 표의 해당 행을 참조한다.

### US-5 (S4/S1) 약기법 리스크를 문장 단위로 걸러낸다
"규제에 민감한 담당자로서, 내 카피의 어느 문장이 일본 薬機法상 못 쓰는 표현인지, 그리고 소구력을 유지한 합법 대체 표현이 무엇인지 알고 싶다. 왜냐하면 광고 정지·회수 리스크를 상장 전에 제거해야 하기 때문."

### US-6 (S1) 근거로 경쟁 벤치마크·리뷰 우려를 본다
"담당자로서, 감이 아니라 일본 상위 제품 실측 표현과 대비해 뭐가 빠졌는지, 일본 口コミ에서 어디가 불신을 살지 알고 싶다."

# 10. 보고용 슬라이드 내보내기 (v3 신규 · 2026-07-16)

발행된 리포트를 **상사 품의용 슬라이드(단일 HTML)** 로 내보낸다. 리포트 화면의 버튼으로 **온디맨드** 생성하며, 진단 파이프라인에는 넣지 않는다(안 쓰는 사람에게 비용·시간을 물리지 않는다).

## 10.1 위상 — 왜 리포트가 있는데 슬라이드인가
블록0(품의용 요약 표지)은 **한 장짜리 증빙**이고, 슬라이드는 **말로 설명하는 자리**용이다. 리포트는 근거를 다 펴 보이지만 결재 회의에서 넘기기엔 길다. 슬라이드는 같은 사실을 7장으로 압축하되 **새 사실을 만들지 않는다**.

## 10.2 형식 (확정)
- **단일 self-contained HTML 1개 파일.** 외부 CSS/JS/폰트/이미지 **의존 0** — 디스크에서 열어 발표·인쇄·PDF 저장. Chart.js 등 CDN 금지.
- 브라우저에서 화살표/클릭 내비게이션(인라인 스크립트). 16:9.
- 디자인 토큰은 `design/DESIGN.md` 코랄 시스템(`#ff6464` CTA · `#d93636` hover/링크).

## 10.3 산출 로직 — 숫자는 LLM이 만들지 않는다 (증거 원칙의 구조적 보장)
| 무엇 | 누가 |
|---|---|
| **모든 수치**(종합점수·Top3·감사 불가/조건부/가능 건수·코퍼스 표본 수·가격) | **코드** — `blocksJson`에서 직접 인용 |
| **슬라이드 목록·순서**(모드별 고정 골격 — v4) | **코드** — 브랜드+제품 7장 / 브랜드 4장 |
| **한국어 품의 카피**(표제·한 줄 요지·결재 포인트) | **LLM 콜⑤** |

> **왜 이렇게 나누나:** LLM이 숫자를 낼 수 있으면 "가짜 수치를 검증으로 걸러내야" 하지만, 애초에 못 내게 하면 **구조적으로 불가능**해진다. 콜②가 LLM 판정 위에서 summary를 코드로 재계산하는 것과 같은 수법(§5.3 결정성 계약).

## 10.4 골격 (모드별 고정 — v4)
- **브랜드+제품 진단 (7장):** `표지` → `결론 한 장` → `점수·Top3` → `약기법 리스크` → `벤치마크 갭` → `비포·애프터 1종` → `다음 단계·비용`
- **브랜드 진단 (4장):** `표지` → `포지셔닝·USP 재정의` → `벤치마크(코퍼스 측)` → `다음 단계·비용`
  - 잠긴 4장을 "측정 못 했습니다"로 채우지 않는다 — 덱은 품의 산출물이다. 결론·점수·리스크·비포애프터 장은 **존재하지 않으므로** 없는 수치를 렌더할 경로 자체가 없다.

품의 문서의 뼈대다: **얼마에 뭘 받았나 → 결론 → 근거 → 위험 → 그래서 뭘 하나.**

## 10.5 콜⑤ 계약
- 입력: `blocksJson` 발췌(모드에 존재하는 블록만) + `tierInput`. **코퍼스 grounding 주입 안 함** — 일본어 카피를 판정하는 콜이 아니고, 코퍼스는 이미 `blocksJson`에 구워져 있다(캐시 프리픽스 낭비 방지). 골격 프리픽스는 모드별 2종(각 모드 안에서 캐시 안정).
- 출력: 슬라이드 키별 `{ heading, lead, bullets[] }`. 스키마 고정 JSON.
- `maxTokens` 4000 · 검증 = 모드의 키 전부 존재 + 카피 비어있지 않음.
- 실패 시 라우트가 500 — **슬라이드 실패가 리포트에 영향을 주지 않는다**(별도 요청).

## 10.6 저장 — 안 함 (재생성)
매 다운로드마다 콜⑤ + 렌더. 저장된 덱은 `blocksJson` 재생성 시 즉시 낡고, HTML blob은 죽은 무게다. **나중에 필요하면 HTML이 아니라 덱 spec(수 KB JSON)을 저장** — 재렌더는 공짜·결정적.

## 10.7 수용 기준
- **AC-10.1** 발행(`published`) 상태에서만 내보낼 수 있다. 그 외 409.
- **AC-10.2** 출력 HTML에 **외부 리소스 참조가 없다**(`src="http`·`href="http`·CDN 0건).
- **AC-10.3** 슬라이드의 모든 수치가 `blocksJson`과 일치한다. LLM 카피에 다른 숫자가 있어도 렌더 결과는 `blocksJson`을 따른다.
- **AC-10.4** 사용자 입력(브랜드명 등)과 LLM 카피가 **HTML 이스케이프**되어 삽입된다.
- **AC-10.5** 같은 입력 → 같은 HTML(렌더러는 순수 함수).
- **AC-10.6** 일본어 구간에 `lang="ja"`가 지정된다.

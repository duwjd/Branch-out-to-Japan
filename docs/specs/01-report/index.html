① 진단 리포트 · 와이어프레임

01-report-ui-기획서.md의 4개 섹션·30행 정의표를 화면으로 옮긴 것이다. 모든 프레임에 기획서 번호(INPUT-02 등)를 달아 문서와 1:1로 대조된다. 기획서에 없는 화면·동작은 넣지 않았다.

작성 2026.07.16 @Calia Kim 비배포 · 기획 검토용 단일 폴더 · 무의존

예시 데이터 고지. 화면 속 종합점수(18/100)·감사 판정·K1~K11 문장·페르소나·일본어 카피는 정본 샘플(../01-report-sample-cica-ampoule.md · 가상 브랜드 HARUON CICA 앰플)에서 가져온 UI 시연용 예시다. 실제 진단 결과가 아니다. 코퍼스·규정 실측 밖의 가상 스탯은 넣지 않았다(증거 원칙).
FLOW

전체 플로우

입력 → 처리 → 리포트 → 슬라이드

파이프라인이 성공하면 곧바로 발행되고, 고객은 리포트를 열람한 뒤 보고용 슬라이드를 내려받는다. 사람이 개입하는 단계는 없다(2026-07-16 검수 제거). 각 노드를 클릭하면 해당 화면으로 이동한다.

INPUT

진단 입력

브랜드 필수 · 제품 선택(콘텐츠 포함) · 두 진단 모드

 → PROCESS

진단 처리

2.5초 폴링 · 브랜드+제품 = LLM 4콜 / 브랜드 = 콜③ 1콜

 → REPORT

리포트 열람

블록 0~9 세로 스크롤 · 발행 배너 · 슬라이드 내보내기

기획서 대조. 노드 = 기획서 섹션. INPUT = # 1. 진단 입력 · PROCESS = # 2. 진단 처리 · REPORT = # 3. 리포트 열람.
2026-07-16: # 4. 검수 · 발행 삭제 — 검수 단계가 제품에서 제거되어 파이프라인 성공 = 발행이 됐다. → ../01-report-spec.md v3 배너

STATE

상태 머신

08-data-flow §3.3

화면은 상태를 그리는 껍데기다. 아래 4개 상태가 화면 분기의 전부이며, 기획서의 각 상태 서술이 여기에 대응한다.

submitted

제출됨

요청 생성 직후. 처리 로딩 출력

 → processing

처리 중

7단계 진행. 2.5초 폴링
published

발행 완료

파이프라인 성공 = 발행. 잡이 직접 세팅(사람 관여 0)

 failed

처리 실패

콜② 실패 = 잡 실패. 감사표 없는 리포트는 발행 X

터미널 상태는 published · failed 2개다. 이 상태에 도달하면 폴링이 멈춘다(PROCESS-01 1a).
2026-07-16: needsReview · rejected 삭제 — 검수 단계 제거. 기존엔 signReport만 published를 세팅했으나 이제 잡이 직접 세팅한다.

OPEN

이 와이어프레임이 그리지 않은 것

기획서에서 (미정)·(추후 기획)·미구현으로 표시한 항목은 화면을 그리지 않았다. 확정 전에 그리면 결정된 것처럼 읽히기 때문이다. 브랜드 진단의 데이터 잠금 상태는 3-report.html#f-brandmode 프레임으로 반영됐다(2026-07-16 시각화 개편과 함께).

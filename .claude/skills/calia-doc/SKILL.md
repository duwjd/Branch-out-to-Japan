# calia-doc

Calia(@Calia Kim)가 젬젬 비디오/젬젬 프로젝트에서 작성해 온 기획서와 동일한 형식·어투로 기획서를 작성하기 위한 스킬이다.

기획서는 화면 또는 기능 단위로 **개조식 표**를 중심으로 서술한다. 서술형 문단은 개요에만 쓰고, 나머지는 표 안의 계층형 불릿으로 정리한다. 문장은 짧고 단정형으로 끝낸다.

## 언제 쓰는가

- 새 기능/화면/플로우를 기획 문서로 정리해 달라는 요청
- 기존 기획서와 같은 양식으로 문서를 작성해 달라는 요청
- 화면 정의, UI 동작, 예외처리, 상태별 동작을 표로 정리하는 작업

## 문서 구조 (이 순서를 지킨다)

```
# {문서 제목}

{변경 이력 표}

# 0. 개요
{서술형 개요 문단}

# 1. {섹션 제목}
{도입 문단 - 선택}

{화면/기능 정의 표}

# 2. {섹션 제목}
...
```

규칙:

- 제목은 문서 최상단에 `# {제목}` 하나. 기능명을 그대로 쓴다. 예: `# 일괄 생성 기능 기획`, `# 영상 생성 플로우 변경 기획서`.
- 문서가 단일 흐름이면 개요 뒤에 바로 정의 표 하나만 둔다(예: 크롭 기획서). 흐름이 여러 갈래거나 케이스가 나뉘면 `# 1.`, `# 2.`로 나눈다.
- 하위 갈래는 `## 2.1.`, `## 2.2.`로 나눈다(예: 에디터 기획의 텍스트/이미지/필터/스티커/레이어).
- 개요가 짧거나 불필요하면 생략할 수 있다. 단, 신규 기능은 개요를 넣는다.

문서는 위 요소(제목 · 변경 이력 · 개요 · 정의 표 / 섹션)로만 구성한다. 이 외의 것은 넣지 않는다.

## 문서에 넣지 않는 것 (중요)

Calia의 실제 기획서에 없는 항목은 절대 추가하지 않는다. 프로젝트에 다른 템플릿이 있어도 따르지 않는다. 특히 다음을 넣지 않는다.

- **릴리즈 계획** 섹션
- **Open Issues** 섹션
- **부록 / 작성 체크리스트** 섹션
- 유저 스토리·시나리오, 유저 플로우 다이어그램 등 별도 섹션
- 목차, 문서 정보 표(변경 이력 표 외 추가 메타 표)
- SVG·이미지 와이어프레임을 임의로 그려 넣기 (이미지 칸은 비워 둔다)
- 완료 후 자기 점검용 체크리스트나 설명 주석 (`<!-- ... -->` 포함)

`references/template.md`는 뼈대를 잡기 위한 채움용 양식이다. `{중괄호}` 자리표시자와 안내 문구는 실제 내용으로 교체하고, 주석·안내 텍스트는 최종 문서에 남기지 않는다.

## 변경 이력 표

문서 제목 바로 아래에 넣는다. 컬럼은 `작성일 | 작성자 | 비고`.

```
| **작성일** | **작성자** | **비고** |
| --- | --- | --- |
| 2026.04.30 | @Calia Kim | 최초 작성 |
| 2026.05.14 | @Calia Kim | [추가] 이미지 업로드 관련 내용 [수정] 로딩 UI 추가 |
```

- 날짜는 `YYYY.MM.DD`.
- 첫 행 비고는 `최초 작성`.
- 이후 변경은 `[추가]`, `[수정]`, `[삭제]` 태그로 시작해 무엇을 바꿨는지 한 줄로 적는다. 한 행에 여러 변경을 이어 쓸 수 있다.
- 신규 문서를 처음 쓰는 경우 오늘 날짜로 `최초 작성` 한 행만 넣는다. 작성자는 `@Calia Kim`.

## 개요 (`# 0. 개요`)

유일하게 서술형 문단으로 쓰는 영역이다. 다음을 담되, 아는 만큼만 짧게 적는다.

- 기능 정의: 이 기능이 무엇인지 한 문장으로.
- 목적/대상: 누구에게 왜 제공하는지(예: B2B 유저 대상).
- 범위·제약: 지원 환경, 기존 플로우와의 차이, 다루지 않는 것(예: "모바일 미지원, PC 기준").

문단도 단정형으로 끝낸다. 예: "일괄 생성 기능은 한 가지 스타일로 여러 개의 프로젝트를 한 번에 생성할 수 있는 기능이다. B2B 유저에게 제공하는 기능으로서, 별도의 피처로 분리된다."

## 화면/기능 정의 표

기획의 본문이다. 컬럼은 `이미지 | 번호 | 내용` 세 개만 둔다. 기존의 제목·비고 컬럼은 두지 않고, 그 내용은 아래 규칙대로 `내용` 칸에 녹인다.

```
| **이미지** | **번호** | **내용** |
| --- | --- | --- |
|  | HOME-00 | **진입** - 메인페이지에서 "일괄 생성" 피처 버튼을 클릭 |
|  | HOME-01 | **모든 화면에 고정 노출되는 전역 내비게이션**1a. gemgem.ip 로고1b. 탭바 접기 - 접혔을 때 각 메뉴의 아이콘만 표시 - 클릭 시 탭바가 접히고 펴짐1c. 탭바 메뉴 - 홈/프로젝트/템플릿으로 메뉴 구성 - 현재 표시하는 메뉴는 하이라이트 처리 |
|  | HOME-02 | **로컬 파일 업로드 영역** - 영역에 파일을 드래그 앤 드롭하여 업로드 - 영역 클릭 시 파일 탐색기 출력 - 업로드 성공 시 업로드 뷰로 이동함 - "업로드에 실패했습니다" 출력 |
|  | HOME-03 | **기타** - 어느 항목에도 안 들어가는 정책·부가 규칙을 모음 |
```

컬럼 규칙:

- **이미지**: 와이어프레임/캡처가 들어갈 자리. 없으면 빈 칸으로 둔다(임의로 채우지 않는다).
- **번호**: `{섹션코드}-{2자리 일련번호}` 형식으로 매긴다. 예: `HOME-01`, `HOME-02`.
    - 섹션코드는 그 섹션(`# 1. 홈`)의 제목을 **영문 대문자**로 옮긴 약칭이다. 예: 홈 → `HOME`, 프로젝트 → `PROJECT`, 업로드 → `UPLOAD`.
    - 일련번호는 `00`부터 두 자리로 0을 채운다. `-00`은 항상 **진입**(어디서 어떻게 들어오는지). 이어서 `-01`, `-02` 순으로 화면/단계를 매긴다.
    - 표의 마지막 번호는 **기타**로 두고, 어느 항목에도 안 들어가는 정책·부가 규칙을 모은다. 기타는 마지막 일련번호를 그대로 이어 쓴다(예: 본문이 `-04`까지면 기타는 `-05`).
- **내용**: 개조식 계층 불릿. 다음 순서로 구성한다.
    - **맨 앞 줄에 그 단계의 제목/한 줄 정의를 볼드로** 쓴다(예: `**모든 화면에 고정 노출되는 전역 내비게이션**`). 별도의 제목 컬럼은 없으므로 제목은 여기 첫 줄에 볼드로 둔다.
    - 한 단계 안에서 갈래가 나뉘면 **하위 번호 `1a`, `1b`, `1c` …** 로 매긴다. 앞 숫자는 그 행의 일련번호(예: `HOME-01` → `1a`/`1b`/`1c`, `HOME-04` → `4a`/`4b`)를 쓰고, 뒤 알파벳은 갈래 순서다. 하위 번호 뒤에 그 갈래의 이름을 이어 쓴다(예: `1b. 탭바 접기`).
    - 하위 번호 아래의 동작·규칙·상태·예외는 불릿(` - `)으로 전개한다.
    - 갈래가 없으면 하위 번호 없이 볼드 정의 뒤에 바로 불릿을 단다.
    - 스펙·수치·포맷·조건·참고 링크(기존 비고에 적던 것)도 해당 불릿에 함께 적는다. 예: `- 글자수 최대 50자`, `- 업로드 가능한 이미지 포맷 : png, jpg, jpeg, webp`, `- 크기 : 10mb 이하`.

## 어투·문체 (가장 중요)

이 스킬의 핵심은 형식보다 **어투**다. Calia의 기획서는 짧고 단정하고 군더더기가 없다.

**단정형 종결.** 모든 서술은 `-다 / -함 / -됨 / -음 / -X` 로 끝낸다. "~할 수 있습니다", "~하게 됩니다" 같은 존댓말·풀어쓰기를 쓰지 않는다.

- 좋음: "클릭 시 파일 업로드 UI 출력", "초기화 버튼 클릭 시 팝업 출력", "실패한 이미지가 없을 때는 실패 개수 출력 X"
- 나쁨: "클릭하시면 파일을 업로드할 수 있는 창이 나타나게 됩니다"

**개조식 계층.** 문장을 문단으로 잇지 않고 불릿으로 쪼갠다. 상위 규칙 아래에 조건·예외를 들여쓰기로 단다.

```
- "생성하기" 버튼을 클릭하여 영상을 생성할 수 있음
    - 크레딧이 부족한 경우 버튼 터치 시 동작은 기존 로직과 동일함
    - 업로드중이거나 업로드 실패한 이미지가 있을 때 버튼 터치 시
        - 해당 이미지 파일이 보이도록 스크롤 이동
```

**짧게.** 한 불릿은 한 가지만 말한다. 뺄 수 있는 단어는 뺀다. 조사·수식어를 최소화한다.

**표기 규칙.**

- 동적 값·변수는 중괄호로: `{사용자가 업로드한 이미지 수}/{최대 업로드 가능 수}`, `{n}개`, `{sec}초`.
- UI에 노출되는 문구·토스트·버튼명은 큰따옴표로: `"파일 {n}개를 업로드하지 못했어요"`, `"추가 업로드"`, `"초기화"`.
- 이미 구현된 동작은 새로 정의하지 말고 참조: "기존과 동일함", "현재 구현된 기능을 사용함", "기존 로직과 동일".
- 확정 사항은 단정형으로 적고, 미확정 사항은 문장 안에 "(미정)", "(추후 기획)"처럼 짧게 표시한다.
- 예외처리·상태별 동작(로딩/실패/빈 상태/성공)을 빠뜨리지 않고 각각 적는다. 예: "이미지 업로드중일 때 : 스피너 표기", "업로드 실패 시 : 실패 및 재업로드 UI 출력".
- 수치·픽셀·포맷 등 스펙은 해당 불릿에 명시(예: `세로 500px 이상 시 스크롤 발생`, `300*300 픽셀`).

## 금지 (불용어·미사여구)

기획서에 감정·홍보·완충 표현을 쓰지 않는다. 사실과 동작만 적는다.

- 미사여구·홍보성: "혁신적인", "매끄러운 경험", "직관적으로", "손쉽게", "간편하게" → 삭제.
- 완충·모호어: "~할 수 있도록 합니다", "~하는 것이 좋습니다", "~일 수도 있습니다"(불확실하면 "(미정)"으로 표기), "기본적으로", "일반적으로" → 삭제하거나 단정형으로 고쳐 쓴다.
- 군더더기 연결어: "그리고", "또한", "이를 통해", "즉" → 불릿 분리로 대체.
- 존댓말 종결(-습니다/-됩니다/-세요) → 단정형으로 통일.

문장을 다 쓴 뒤 "이 단어를 빼도 뜻이 같은가?"를 점검해 뺄 수 있는 건 뺀다.

## 작성 절차

1. 요청에서 기능명, 대상 화면/단계, 동작, 예외를 파악한다. 정보가 부족하면 채우지 말고 "(미정)" 또는 "(확인 필요)"로 남긴다.
2. 문서 제목 → 변경 이력(오늘 날짜, 최초 작성) → 개요 → 정의 표 순으로 뼈대를 잡는다.
3. 섹션코드를 정하고(섹션 제목의 영문 대문자 약칭), 표의 번호를 진입(`-00`)부터 기타(마지막)까지 `{섹션코드}-{2자리}` 형식으로 채운다. 각 행의 내용은 볼드 제목 → (필요 시) 하위 번호 `1a`/`1b`/`1c` → 불릿 순으로 개조식·단정형으로 전개한다.
4. 상태별 동작과 예외처리를 각 단계에 반영한다.
5. 아래 체크리스트를 통과하는지 확인한다.

`references/template.md`에 빈 템플릿이 있다. 새 문서는 이를 복사해 채운다.

## 완료 체크리스트 (내부 점검용 — 문서에 쓰지 않는다)

- [ ] 제목·변경 이력·개요·정의 표 순서가 맞다.
- [ ] 릴리즈 계획·Open Issues·작성 체크리스트·유저 스토리 등 참조 문서에 없는 섹션이 없다.
- [ ] 안내 주석(`<!-- -->`)이나 자리표시자 `{}`가 최종 문서에 남아 있지 않다(변수 표기는 예외).
- [ ] 변경 이력에 오늘 날짜와 작성자(@Calia Kim)가 있다.
- [ ] 모든 서술이 단정형(-다/-함/-됨/-음)으로 끝난다. 존댓말이 없다.
- [ ] 정의 표 컬럼이 `이미지 | 번호 | 내용` 세 개다.
- [ ] 번호가 `{섹션코드}-{2자리}` 형식이고, `-00`은 진입, 마지막은 기타다.
- [ ] 각 행 내용의 첫 줄이 볼드 제목이고, 갈래는 `1a`/`1b`/`1c` 하위 번호로 매겨졌다.
- [ ] 내용이 개조식 계층 불릿이다. 서술형 문단은 개요에만 있다.
- [ ] 변수는 `{}`, UI 문구는 `""`로 표기했다.
- [ ] 상태별 동작(로딩/실패/빈 상태/성공)과 예외처리를 적었다.
- [ ] 미사여구·완충어·불용어가 없다.
- [ ] 확정/미정이 구분돼 있다.

## 대시보드 (앱 홈)

- **목적**: 온보딩 완료 후 진입하는 앱 홈이자 **3축 허브 뷰**. IA §4 명시 3요소 = **브랜드 자산 요약 · 진행중 작업 · 다음 단계 CTA**.
- **연결**: in ← 온보딩(브랜드 프로필 등록) / 로그인 · out → ① 진단 · ② 스튜디오 · ③ 운영 · 품의용 PDF.
- **컨텍스트**: 공통 엔티티 **브랜드 프로필** 위에서 3축이 자산 공유(약기법 엔진 ①↔② 재사용).

### 요소 인벤토리
| 영역 | 요소 | 역할 | 컴포넌트 |
|---|---|---|---|
| 상단바 | **브랜드 프로필 스위처**(신규) | 작업 대상 브랜드 컨텍스트 고정·전환 | Button+listbox(`aria-haspopup`) |
| 상단바 | 3축 내비(대시보드/①/②/③) + 품의용 PDF + Avatar | 축 전환·결재 산출·계정 | nav·Button·Avatar |
| 요약 | KPI 타일 4 | 리포트 수·생성 자산·재점검 점수·매칭 | Stat |
| 3축 진입 | 축 카드 + **성숙도 배지** | 이용가능/썸네일 우선/준비 중 | Card·Badge |
| 진행중 작업 | 리포트 v1(재점검)·썸네일 그리드 | 최근 자산 | Card·asset·thumbs |
| 운영 | 일본 기업 매칭 | 매칭 리스트 | Card·match·Badge |
| 다음 단계 | CTA 패널(② 이어가기 + 품의 PDF) | 고정가 퍼널 out-link | Card(tint)·Button |

### 성숙도 배지 (실제 개발 상태와 일치 · IA §6)
| 축 | 배지 | 근거 |
|---|---|---|
| ① 진단 리포트 | `이용 가능`(success) | 스펙 확정 v2 |
| ② 마케팅 스튜디오 | `썸네일 우선`(warning) | 썸네일만 파이프라인 확정 |
| ③ 운영 | `준비 중`(neutral) | 방향만 · 성과 판별 검토 |

### 상태 (DESIGN.md §14 · [반영]=예시 있음)
- **Empty** [반영]: 첫 방문(자산 0) → "아직 진단 리포트가 없어요 · 첫 진단 시작"(이유+액션).
- **Loading** [반영]: 스켈레톤(카드 치수 유지).
- **Error(네트워크)**: 대시보드 로드 실패 → 중앙 heading+retry `[디자이너]`.
- **Focus**: 스위처·축 카드·버튼 `:focus-visible` 링.

### IA 반영 · 신규 요소 (프로토타입 #ops 대비)
1. **브랜드 프로필 스위처** — 상단바에 신설(IA §5, #ops엔 없던 요소).
2. **3축 성숙도 배지** — 화면이 실제 상태를 오인시키지 않도록.
3. **온보딩→리포트 프리필** — 대시보드가 온보딩 프로필과 리포트 입력을 잇는 자리(중복 입력 제거).
4. **품의 PDF 재접근** — 상단바 + 다음 단계 패널 양쪽(품의형 사용자 결재 동선, IA §3).
5. **카테고리 택소노미 정본** — 프로필 표기 "스킨케어 / スキンケア"(리포트 엔진 키, `public-onboarding-spec.md §0`).

### 데이터 모델 연결 (IA §7)
```
브랜드 프로필 ├ 상품 └ 채널상정(Qoo10/라쿠텐/아마존JP) └ 상세 카피(K1…Kn)
            ├ 생성 자산 ├ 진단 리포트(블록0~9·PDF) └ 썸네일(스타일 A~H·플랫폼별)
            └ 시즌 캘린더(메가와리 3·6·9·11월·코프레·UV 4~7월)
```
→ KPI 타일·자산 섹션이 이 3자산군(리포트/썸네일/시즌·매칭)을 반영.

## 디자이너 결정 필요
- 브랜드 스위처 드롭다운 UX · **다중 브랜드 지원 범위**(1개 전제 vs 다수)
- **결제/구독 상태 표기**(1회 Report vs 월 구독 Growth) — 커머셜 미확정(IA §10, `ux-review.md` C4)
- 모바일 앱 셸(햄버거·하단 탭) 구조
- 매칭/성과판별 실제 스펙(③ 운영 스펙 미착수)
